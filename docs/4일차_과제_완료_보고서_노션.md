## 과제 1 — User API 설계하기

### 과제 목표

사용자 요구사항을 분석하여 구현 전에 사용할 API 명세서를 작성하고, Git branch 및 Pull Request 과정을 거쳐 `main` 브랜치에 반영하는 것이 목표였다.

### 설계한 API

| Method | Endpoint | 기능 |
| --- | --- | --- |
| `POST` | `/api/v1/users/signup` | 회원가입 |
| `POST` | `/api/v1/users/login` | 로그인 |
| `POST` | `/api/v1/users/refresh` | Access Token 재발급 |
| `POST` | `/api/v1/users/logout` | 로그아웃 |
| `GET` | `/api/v1/users/me` | 내 정보 조회 |
| `PATCH` | `/api/v1/users/me` | 내 정보 수정 |
| `PATCH` | `/api/v1/users/me/password` | 비밀번호 변경 |
| `DELETE` | `/api/v1/users/me` | 회원 탈퇴 |
| `GET` | `/api/v1/admin/users` | 관리자 사용자 목록 조회 |
| `PATCH` | `/api/v1/admin/users/role` | 관리자 사용자 권한 수정 |

### 설계 문서에 포함한 내용

- 요구사항과 API의 매핑
- Endpoint, HTTP Method, 인증 및 권한
- Request·Response Schema와 JSON 예시
- 성공·예외 상태 코드
- 이메일·비밀번호·전화번호 검증 규칙
- User 모델 필드 매핑
- JWT 및 비밀번호 보안 정책
- Router → Schema → Service → Repository → Model 처리 흐름
- 구현 및 테스트 체크리스트

## 과제 2 — User API 작성하기

### 과제 목표

과제 1에서 작성한 명세를 기준으로 User API 전체를 구현하고, Swagger UI와 자동 테스트로 정상 동작을 확인한 뒤 Pull Request를 통해 `main` 브랜치에 반영하는 것이 목표였다.

### 구현 구조

| 계층 | 주요 파일 | 역할 |
| --- | --- | --- |
| Router | `app/apis/users.py`, `admin_users.py` | 요청 수신, 인증 의존성, 상태 코드와 응답 처리 |
| Schema | `app/schemas/user.py` | Pydantic 요청값 검증과 응답 구조 정의 |
| Service | `app/services/user_service.py` | 중복 검사, 권한, 비밀번호 등 비즈니스 규칙 처리 |
| Repository | `app/repositories/user_repository.py` | SQLAlchemy 사용자 조회·생성·수정 |
| Security | `app/core/security.py` | Argon2 해싱과 JWT 생성·검증 |
| Dependency | `app/core/dependencies.py` | 현재 사용자와 관리자 인증 처리 |

### 주요 구현 내용

- User API 10개 구현
- Pydantic `AfterValidator` 및 Field Validator 적용
- 이메일 형식과 중복 검증
- 전화번호 정규화와 중복 검증
- 비밀번호 대문자·소문자·숫자·특수문자 검증
- `pwdlib` Argon2 비밀번호 해싱
- JWT Access Token 및 Refresh Token 적용
- Refresh Token을 HttpOnly Cookie로 전달
- 사용자와 관리자 권한 분리
- 회원 탈퇴 시 `is_active=false` 비활성화
- 비밀번호와 `hashed_password` 응답 제외
- 프런트엔드 Enum과 DB Enum 호환 처리

### 테스트 및 Swagger 검증

- Swagger UI `/docs`: `200 OK`
- OpenAPI User API operation: 10개
- 자동 테스트: `3 passed`
- 회원가입 → 로그인 → 갱신 → 조회 → 수정 → 비밀번호 변경 → 관리자 권한 변경 → 탈퇴 전체 흐름 통과
- 인증 없음, 잘못된 로그인, 중복 이메일, 권한 부족, 잘못된 입력 예외 테스트 통과
- 테스트는 격리된 SQLite DB를 사용하여 실제 MySQL 데이터에 영향을 주지 않음
- `compileall`, `uv lock --check`, `git diff --check` 통과

## 학습 회고

이번 과제를 통해 API 구현 전에 요구사항을 Endpoint, 요청값, 응답값, 상태 코드와 예외 조건으로 구체화하는 과정이 중요하다는 점을 배웠다. 명세를 먼저 작성하니 구현해야 할 범위와 각 계층의 책임을 명확하게 나눌 수 있었다.

FastAPI 구현에서는 Router에 모든 로직을 작성하지 않고 Schema, Service, Repository로 역할을 분리했다. 이를 통해 입력값 검증, 비즈니스 규칙, DB 접근을 독립적으로 관리하고 테스트하기 쉬운 구조를 만들 수 있었다.

인증 구현 과정에서는 비밀번호를 평문으로 저장하지 않고 Argon2로 해싱했으며, Access Token과 Refresh Token의 역할을 구분했다. 또한 일반 사용자와 관리자 권한을 분리하고 비활성화된 계정의 접근을 차단하는 흐름을 학습했다.

마지막으로 정상 요청뿐 아니라 잘못된 비밀번호, 중복 데이터, 인증 실패, 권한 부족과 같은 예외 상황도 자동 테스트로 확인했다. 코드 작성 후 Swagger UI, OpenAPI, 자동 테스트, 잠금 파일과 Git 변경 범위까지 검증하면서 기능 구현과 검증 기록을 함께 남기는 과정의 중요성을 확인했다.
