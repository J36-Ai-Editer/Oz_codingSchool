# 9일차 동시성 문제 해결을 위한 아키텍처 설계

## 1. 문서 목적

이 문서는 FastAPI와 Redis를 이용해 Event-Driven Architecture를 설계하는 방법을 정리한 학습 문서다.

현재 프로젝트는 환자 정보, 진료기록, X-Ray 이미지 업로드, AI 분석 결과 조회 기능을 포함한다. 이 중 X-Ray 이미지 분석처럼 시간이 오래 걸리는 작업을 HTTP 요청 안에서 바로 처리하면 사용자는 응답을 오래 기다리게 되고, 동시에 여러 요청이 들어올 경우 서버 부하와 동시성 문제가 발생할 수 있다.

따라서 FastAPI는 요청을 빠르게 받고, Redis는 이벤트를 전달하거나 작업 대기열 역할을 맡도록 분리하는 구조를 학습한다.

## 2. Event-Driven Architecture란?

Event-Driven Architecture는 어떤 일이 발생했을 때 그 사실을 이벤트로 발행하고, 해당 이벤트를 필요로 하는 처리자가 나중에 처리하는 구조다.

예를 들어 사용자가 X-Ray 이미지를 업로드하면 서버가 바로 AI 분석까지 끝낼 때까지 기다리는 대신 다음과 같이 처리할 수 있다.

```text
X-Ray 이미지 업로드
→ xray.uploaded 이벤트 발행
→ AI 분석 Worker가 이벤트 수신
→ 분석 실행
→ 분석 결과 저장
→ analysis.completed 이벤트 발행
→ 사용자는 결과 조회 API 또는 알림으로 확인
```

이 구조의 핵심은 요청을 받은 서버와 실제 무거운 작업을 처리하는 Worker를 분리하는 것이다.

## 3. 기존 동기 처리 방식의 문제점

기존 동기 처리 방식은 API 요청 하나가 모든 작업을 끝낸 뒤 응답을 반환한다.

```text
Client
→ FastAPI
→ 이미지 저장
→ AI 분석 실행
→ DB 저장
→ Response
```

이 방식은 단순하지만 다음 문제가 있다.

| 문제 | 설명 |
| --- | --- |
| 응답 지연 | AI 분석이 오래 걸리면 사용자가 계속 기다려야 한다. |
| 서버 부하 증가 | 동시에 여러 분석 요청이 들어오면 FastAPI 서버가 무거운 작업까지 처리해야 한다. |
| 장애 전파 | AI 분석 중 오류가 나면 API 요청 자체가 실패할 수 있다. |
| 재처리 어려움 | 처리 중 실패한 작업을 다시 실행하기 어렵다. |
| 확장성 부족 | API 서버와 분석 처리 서버를 독립적으로 늘리기 어렵다. |

## 4. FastAPI와 Redis의 역할

FastAPI와 Redis를 함께 사용하면 API 서버와 작업 처리 흐름을 분리할 수 있다.

| 구성 요소 | 역할 |
| --- | --- |
| FastAPI | 클라이언트 요청 수신, 데이터 검증, DB 저장, 이벤트 발행 |
| Redis Pub/Sub | 실시간 알림처럼 유실되어도 큰 문제가 없는 이벤트 전달 |
| Redis Streams | 작업 대기열처럼 이벤트를 저장하고 Worker가 안정적으로 처리 |
| Worker | Redis에서 이벤트를 읽어 실제 비동기 작업 수행 |
| Database | 요청 데이터, 작업 상태, 분석 결과 저장 |

FastAPI의 `BackgroundTasks`는 응답 이후 간단한 후처리를 실행할 때 사용할 수 있다. 하지만 무거운 작업을 여러 서버나 여러 프로세스에서 안정적으로 처리해야 한다면 Redis 같은 메시지 브로커를 사용하는 구조가 더 적합하다.

## 5. Redis Pub/Sub과 Redis Streams 비교

Redis는 이벤트 기반 설계에서 크게 Pub/Sub과 Streams 방식으로 활용할 수 있다.

| 구분 | Redis Pub/Sub | Redis Streams |
| --- | --- | --- |
| 주요 목적 | 실시간 메시지 전달 | 이벤트 저장 및 작업 큐 |
| 메시지 저장 | 저장하지 않음 | Stream에 저장 |
| 수신자 장애 시 | 메시지 유실 가능 | 나중에 다시 읽기 가능 |
| 재처리 | 어려움 | 가능 |
| Consumer Group | 없음 | 지원 |
| 적합한 상황 | 실시간 알림, WebSocket 브로드캐스트 | 이미지 분석, 이메일 발송, 비동기 작업 처리 |

Redis 공식 문서에 따르면 Pub/Sub은 at-most-once 전달 방식이다. 즉, 메시지가 한 번 전달되거나 전달되지 않을 수 있으며, 구독자가 장애 상태라면 메시지가 사라질 수 있다. 반면 Redis Streams는 메시지를 저장하고 Consumer Group을 통해 여러 Worker가 나누어 처리할 수 있어 작업 큐에 더 적합하다.

현재 프로젝트의 X-Ray AI 분석처럼 반드시 처리되어야 하는 작업은 Pub/Sub보다 Redis Streams가 더 적합하다.

## 6. 프로젝트 적용 대상

현재 프로젝트에서 Event-Driven Architecture를 적용하기 좋은 기능은 다음과 같다.

| 기능 | 이벤트 예시 | 처리 방식 |
| --- | --- | --- |
| X-Ray 이미지 업로드 | `xray.uploaded` | Redis Streams |
| AI 분석 요청 | `analysis.requested` | Redis Streams |
| AI 분석 완료 | `analysis.completed` | Redis Streams 또는 Pub/Sub |
| 분석 실패 | `analysis.failed` | Redis Streams |
| 사용자 알림 | `notification.created` | Pub/Sub 또는 Streams |

가장 우선 적용할 대상은 X-Ray 이미지 업로드 이후 AI 분석 작업이다.

## 7. 제안 아키텍처

### 7.1 전체 구조

```text
Client
  ↓
FastAPI API Server
  ↓
Database에 진료기록과 이미지 정보 저장
  ↓
Redis Streams에 analysis.requested 이벤트 발행
  ↓
AI Analysis Worker가 이벤트 소비
  ↓
AI 분석 실행
  ↓
Database에 분석 결과 저장
  ↓
analysis.completed 이벤트 발행
  ↓
Client가 결과 조회 또는 알림 확인
```

### 7.2 구성 요소별 책임

| 구성 요소 | 책임 |
| --- | --- |
| API Server | 요청 검증, 인증·권한 확인, 파일 저장, 작업 생성, 이벤트 발행 |
| Redis Stream | 분석 요청 이벤트 저장 |
| Worker | 분석 요청 이벤트 소비, AI 모델 실행, 결과 저장 |
| DB | 진료기록, X-Ray 이미지, 분석 상태와 결과 저장 |
| Client | 분석 요청 후 상태 조회 또는 결과 확인 |

## 8. 이벤트 흐름 예시

### 8.1 진료기록 등록 및 분석 요청

사용자가 진료기록과 X-Ray 이미지를 등록하면 FastAPI는 먼저 파일과 기본 정보를 저장한다.

```text
POST /api/v1/patients/{patient_id}/medical-records
```

처리 순서는 다음과 같다.

1. 사용자 인증과 권한을 확인한다.
2. 환자 존재 여부를 확인한다.
3. 진료기록과 X-Ray 이미지 정보를 DB에 저장한다.
4. 분석 상태를 `PENDING`으로 저장한다.
5. Redis Streams에 `analysis.requested` 이벤트를 발행한다.
6. 클라이언트에는 `202 Accepted` 또는 `201 Created`로 빠르게 응답한다.

이벤트 데이터 예시는 다음과 같다.

```json
{
  "event_type": "analysis.requested",
  "xray_image_id": 1,
  "medical_record_id": 10,
  "patient_id": 3,
  "image_path": "media/xray/2026/07/sample.png",
  "requested_by": 5,
  "created_at": "2026-07-24T10:00:00"
}
```

### 8.2 Worker 분석 처리

Worker는 Redis Streams에서 이벤트를 읽고 분석을 수행한다.

```text
Redis Streams
→ Worker
→ AI Model
→ Database
```

처리 순서는 다음과 같다.

1. Worker가 `analysis.requested` 이벤트를 읽는다.
2. X-Ray 이미지 파일을 불러온다.
3. AI 모델로 분석을 실행한다.
4. 분석 결과를 DB에 저장한다.
5. 분석 상태를 `COMPLETED`로 변경한다.
6. 실패하면 상태를 `FAILED`로 변경하고 실패 사유를 저장한다.
7. 처리가 완료된 이벤트는 ACK 처리한다.

## 9. 상태 관리 설계

비동기 작업은 한 번 요청했다고 바로 끝나지 않기 때문에 작업 상태를 DB에 저장해야 한다.

| 상태 | 의미 |
| --- | --- |
| `PENDING` | 분석 요청이 생성되었지만 아직 처리되지 않음 |
| `PROCESSING` | Worker가 분석을 진행 중 |
| `COMPLETED` | 분석이 정상 완료됨 |
| `FAILED` | 분석 중 오류 발생 |

분석 결과 조회 API는 현재 상태를 함께 반환한다.

```json
{
  "analysis_result_id": 1,
  "xray_image_id": 1,
  "status": "COMPLETED",
  "result": "정상",
  "confidence": 0.92,
  "created_at": "2026-07-24T10:01:00"
}
```

분석이 아직 끝나지 않았다면 다음과 같이 응답할 수 있다.

```json
{
  "status": "PROCESSING",
  "message": "AI 분석이 진행 중입니다."
}
```

## 10. 동시성 문제 해결 방식

### 10.1 API 서버 부하 분리

FastAPI 서버는 요청을 받고 이벤트를 발행하는 역할까지만 수행한다. 무거운 AI 분석은 Worker가 처리하므로 API 서버의 응답 시간이 짧아진다.

### 10.2 작업 순서와 중복 처리

Redis Streams는 이벤트를 Stream에 저장한다. Worker가 여러 개여도 Consumer Group을 사용하면 하나의 이벤트를 한 Worker가 담당하도록 나누어 처리할 수 있다.

단, 네트워크 오류나 Worker 장애로 같은 이벤트가 다시 처리될 수 있으므로 멱등성을 고려해야 한다.

멱등성이란 같은 요청이 여러 번 실행되어도 결과가 한 번 실행된 것과 같도록 만드는 것을 의미한다.

예를 들어 `xray_image_id`에 대해 이미 완료된 분석 결과가 있다면 Worker는 중복 분석을 하지 않고 처리 완료로 판단할 수 있다.

### 10.3 실패한 작업 재처리

Worker가 분석 도중 실패하면 DB 상태를 `FAILED`로 저장하고, 필요하면 재시도 횟수를 함께 관리한다.

재시도 정책 예시는 다음과 같다.

| 항목 | 기준 |
| --- | --- |
| 최대 재시도 횟수 | 3회 |
| 재시도 대상 | 일시적 파일 접근 오류, 모델 서버 일시 장애 |
| 재시도 제외 | 잘못된 이미지 형식, 삭제된 파일 |
| 최종 실패 처리 | `FAILED` 상태와 실패 사유 저장 |

## 11. FastAPI 설계 예시

아래 코드는 실제 구현 방향을 이해하기 위한 간단한 예시다.

```python
from fastapi import APIRouter, Depends, status

router = APIRouter(prefix="/api/v1")


@router.post(
    "/patients/{patient_id}/medical-records",
    status_code=status.HTTP_202_ACCEPTED,
)
async def create_medical_record(patient_id: int):
    # 1. 환자 조회
    # 2. 진료기록 저장
    # 3. X-Ray 이미지 저장
    # 4. 분석 상태 PENDING 저장
    # 5. Redis Stream에 analysis.requested 이벤트 발행
    return {
        "message": "진료기록이 등록되었고 AI 분석이 요청되었습니다.",
        "analysis_status": "PENDING",
    }
```

Redis Streams에 이벤트를 발행하는 예시는 다음과 같다.

```python
async def publish_analysis_requested(redis, event: dict):
    await redis.xadd(
        "analysis:requested",
        event,
    )
```

Worker는 다음과 같은 흐름으로 이벤트를 처리할 수 있다.

```python
async def analysis_worker(redis):
    while True:
        events = await redis.xreadgroup(
            groupname="analysis-workers",
            consumername="worker-1",
            streams={"analysis:requested": ">"},
            count=1,
            block=5000,
        )

        for stream_name, messages in events:
            for message_id, data in messages:
                try:
                    # AI 분석 실행
                    # 분석 결과 DB 저장
                    await redis.xack(
                        "analysis:requested",
                        "analysis-workers",
                        message_id,
                    )
                except Exception:
                    # 실패 상태 저장 또는 재시도 처리
                    pass
```

## 12. API 응답 전략

AI 분석처럼 시간이 오래 걸리는 작업은 요청 시점에 결과를 바로 반환하지 않는다.

대신 요청이 정상 접수되었음을 알려주고, 클라이언트가 별도 API로 상태를 확인하게 한다.

| 상황 | 응답 코드 | 설명 |
| --- | --- | --- |
| 작업 요청 접수 | `202 Accepted` | 분석 작업이 큐에 들어감 |
| 진료기록 생성 완료 | `201 Created` | DB 저장까지 완료됨 |
| 분석 진행 중 | `200 OK` | 상태가 `PENDING` 또는 `PROCESSING` |
| 분석 완료 | `200 OK` | 결과 반환 |
| 분석 실패 | `200 OK` 또는 `500` | 실패 상태와 사유 반환 |

현재 프로젝트에서는 진료기록 생성 자체는 완료되므로 `201 Created`를 사용하고, 응답 Body에 분석 상태를 `PENDING`으로 포함하는 방식도 가능하다. 분석 요청을 별도 API로 분리한다면 `202 Accepted`가 더 자연스럽다.

## 13. 장점

FastAPI와 Redis 기반 Event-Driven Architecture를 적용하면 다음 장점이 있다.

| 장점 | 설명 |
| --- | --- |
| 응답 속도 개선 | 무거운 AI 분석을 기다리지 않고 빠르게 응답할 수 있다. |
| 장애 격리 | AI 분석 Worker가 실패해도 API 서버 전체 장애로 이어지지 않는다. |
| 확장성 향상 | 요청이 많아지면 Worker 수를 늘려 처리량을 높일 수 있다. |
| 재처리 가능 | Redis Streams를 사용하면 실패한 이벤트를 다시 처리할 수 있다. |
| 역할 분리 | API 서버와 분석 처리 로직의 책임이 명확해진다. |

## 14. 주의할 점

Event-Driven Architecture는 장점이 많지만 구조가 복잡해지는 만큼 주의할 점도 있다.

| 주의 사항 | 설명 |
| --- | --- |
| 이벤트 유실 | Pub/Sub은 구독자가 없거나 장애 상태면 메시지가 사라질 수 있다. |
| 중복 처리 | Worker 장애나 재시도 과정에서 같은 이벤트가 두 번 처리될 수 있다. |
| 상태 불일치 | DB 저장은 성공했지만 이벤트 발행이 실패하는 상황을 고려해야 한다. |
| 모니터링 필요 | 대기 중인 이벤트 수, 실패 횟수, Worker 상태를 확인해야 한다. |
| 트랜잭션 경계 | DB 작업과 Redis 이벤트 발행을 하나의 트랜잭션처럼 다루기 어렵다. |

## 15. 현재 프로젝트에 맞춘 설계 결론

현재 프로젝트에서는 다음 방향이 적합하다.

1. X-Ray 이미지 업로드와 진료기록 등록은 FastAPI가 담당한다.
2. AI 분석은 HTTP 요청 안에서 바로 실행하지 않고 Redis Streams에 이벤트를 등록한다.
3. 별도 Worker가 Redis Streams에서 이벤트를 읽어 AI 분석을 수행한다.
4. 분석 상태는 DB에 `PENDING`, `PROCESSING`, `COMPLETED`, `FAILED`로 저장한다.
5. 사용자는 분석 결과 조회 API를 통해 현재 상태와 결과를 확인한다.
6. 실시간 알림이 필요하면 분석 완료 시 Pub/Sub 또는 WebSocket을 추가로 사용할 수 있다.

즉, 현재 프로젝트의 핵심 설계는 다음과 같이 정리할 수 있다.

```text
FastAPI = 요청 접수와 이벤트 발행
Redis Streams = 안정적인 비동기 작업 큐
Worker = AI 분석 실행
Database = 상태와 결과 저장
```

## 16. 학습한 내용

이번 학습을 통해 단순히 API를 만드는 것과 실제 서비스에서 안정적으로 작업을 처리하는 것은 다르다는 점을 알 수 있었다.

특히 AI 분석처럼 시간이 오래 걸리는 작업은 API 서버가 직접 끝까지 처리하기보다 이벤트로 분리하는 것이 더 안정적이다. Redis Pub/Sub은 실시간 알림에는 적합하지만 메시지 유실 가능성이 있으므로, 반드시 처리되어야 하는 분석 작업에는 Redis Streams가 더 적합하다는 점도 확인했다.

Event-Driven Architecture는 구조가 조금 복잡해지지만, 동시 요청이 많아지는 상황에서 서버 부하를 줄이고 작업을 안정적으로 처리할 수 있는 설계 방식이다.
