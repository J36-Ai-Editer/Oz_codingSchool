# 3일차 프로젝트 들어보기

## 1. 프로젝트 개요

이번 프로젝트는 FastAPI를 기반으로 웹 API를 만들고, MySQL 데이터베이스와 연결하는 구조를 가진다. Python 패키지 관리는 `uv`를 사용하고, ORM은 SQLAlchemy, 데이터베이스 마이그레이션은 Alembic을 사용한다.

현재 프로젝트의 핵심 구조는 다음과 같다.

```text
AH_web_development_assignment/
├── app/
│   ├── apis/
│   ├── core/
│   │   └── db/
│   ├── models/
│   ├── repositories/
│   ├── schemas/
│   ├── services/
│   └── main.py
├── alembic/
├── static/
├── docker-compose.yml
├── pyproject.toml
└── uv.lock
```

이 구조는 API 계층, 서비스 계층, 데이터 접근 계층, 데이터 모델 계층을 나누기 위한 형태다. 처음에는 파일이 비어 있거나 예제 코드만 있을 수 있지만, 기능이 많아질수록 계층을 나누는 것이 유지보수에 유리하다.

## 2. 디렉터리별 역할

### 2.1 app/core/

`app/core/`는 애플리케이션 전체에서 공통으로 사용하는 설정과 기반 코드를 두는 디렉터리다.

주로 다음 내용을 작성한다.

- 환경변수와 설정 관리
- 데이터베이스 연결 설정
- 인증, 보안, 공통 유틸
- 앱 전역에서 재사용되는 기반 코드

현재 프로젝트에서는 `app/core/config.py`와 `app/core/db/`가 핵심이다.

### 2.2 app/models/

`app/models/`는 SQLAlchemy ORM 모델을 작성하는 디렉터리다. ORM 모델은 데이터베이스 테이블을 Python 클래스로 표현한 것이다.

예를 들어 회원 테이블이 필요하면 `User` 모델을 작성하고, 환자 테이블이 필요하면 `Patient` 모델을 작성한다.

작성 예시는 다음과 같다.

```python
from sqlalchemy import String, Integer
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base


class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    name: Mapped[str] = mapped_column(String(30), nullable=False)
    email: Mapped[str] = mapped_column(String(100), unique=True, nullable=False)
```

모델을 작성한 뒤에는 `app/models/__init__.py`에서 import 해야 Alembic이 모델 변경사항을 인식하기 쉽다.

```python
from app.models.user import User
```

### 2.3 app/repositories/

`app/repositories/`는 데이터베이스에 직접 접근하는 코드를 작성하는 디렉터리다.

API 함수 안에서 바로 SQLAlchemy 쿼리를 작성하면 코드가 금방 복잡해진다. 그래서 DB 조회, 생성, 수정, 삭제 로직은 repository로 분리한다.

예시는 다음과 같다.

```python
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.models.user import User


async def get_user_by_email(db: AsyncSession, email: str):
    result = await db.execute(select(User).where(User.email == email))
    return result.scalar_one_or_none()
```

Repository는 보통 다음 역할을 한다.

- 특정 데이터 조회
- 데이터 생성
- 데이터 수정
- 데이터 삭제
- SQLAlchemy query 작성

### 2.4 app/schemas/

`app/schemas/`는 Pydantic 스키마를 작성하는 디렉터리다.

스키마는 API 요청과 응답의 데이터 구조를 정의한다. 예를 들어 회원가입 요청에는 이름, 나이, 이메일, 비밀번호가 필요하고, 응답에는 비밀번호를 제외한 회원 정보만 내려줘야 한다. 이런 구조를 schema로 표현한다.

예시는 다음과 같다.

```python
from pydantic import BaseModel, EmailStr, Field


class UserCreate(BaseModel):
    name: str = Field(min_length=2, max_length=10)
    age: int = Field(ge=14)
    email: EmailStr
    password: str = Field(min_length=8, max_length=20)


class UserResponse(BaseModel):
    id: int
    name: str
    age: int
    email: EmailStr
```

Schema는 다음 역할을 한다.

- 요청 데이터 검증
- 응답 데이터 형태 정의
- API 문서 자동 생성에 필요한 타입 정보 제공
- DB 모델과 외부 응답 구조 분리

### 2.5 app/services/

`app/services/`는 비즈니스 로직을 작성하는 디렉터리다.

Repository가 단순히 DB 작업을 담당한다면, Service는 서비스 규칙을 담당한다. 예를 들어 이메일 중복 여부 확인, 비밀번호 정책 검사, 권한 확인, 여러 repository를 조합한 작업 등이 여기에 들어간다.

예시는 다음과 같다.

```python
from sqlalchemy.ext.asyncio import AsyncSession

from app.repositories.user_repository import get_user_by_email


async def check_duplicate_email(db: AsyncSession, email: str):
    user = await get_user_by_email(db, email)
    if user:
        raise ValueError("중복된 email은 사용할 수 없습니다.")
```

Service는 보통 다음 역할을 한다.

- 핵심 비즈니스 규칙 처리
- 여러 repository 함수 조합
- 예외 상황 판단
- API가 너무 많은 책임을 갖지 않도록 분리

### 2.6 app/apis/

`app/apis/`는 FastAPI 라우터를 작성하는 디렉터리다.

API 계층은 URL, HTTP method, 요청 schema, 응답 schema, dependency를 연결한다. 실제 비즈니스 로직은 service로 넘기고, DB 접근은 repository가 담당하게 만드는 것이 좋다.

현재 `app/apis/practice_apis.py`에는 연습용 회원 API가 작성되어 있다. 다만 이 파일은 내부에서 `FastAPI()` 앱을 새로 만들고 있으므로, 실제 프로젝트 구조에서는 `APIRouter`를 사용하는 방식이 더 적절하다.

권장 예시는 다음과 같다.

```python
from fastapi import APIRouter, Depends
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.db.databases import async_get_db
from app.schemas.user import UserCreate

router = APIRouter(prefix="/api/v1/users", tags=["users"])


@router.post("")
async def create_user(
    payload: UserCreate,
    db: AsyncSession = Depends(async_get_db),
):
    return {"message": "회원 생성"}
```

그리고 `app/main.py`에서 라우터를 등록한다.

```python
from app.apis.user_api import router as user_router

app.include_router(user_router)
```

## 3. 주요 파일 역할

### 3.1 app/main.py

`app/main.py`는 FastAPI 애플리케이션의 시작점이다.

현재 이 파일은 다음 역할을 한다.

- `FastAPI()` 앱 생성
- `static/` 디렉터리 생성 및 정적 파일 서빙
- `media/` 디렉터리 생성 및 업로드 파일 서빙
- `/healthcheck` API 제공
- `/` 요청 시 `static/index.html` 반환
- 프론트엔드 라우팅을 위한 catch-all 처리

현재 구조에서 API 라우터를 추가하려면 `main.py`에 다음과 같이 연결해야 한다.

```python
from app.apis.practice_apis import router as practice_router

app.include_router(practice_router)
```

단, 이를 위해 `practice_apis.py`도 `app = FastAPI()`가 아니라 `router = APIRouter()` 구조로 바꾸는 것이 좋다.

### 3.2 app/core/config.py

`app/core/config.py`는 환경설정을 관리하는 파일이다.

현재 프로젝트는 `pydantic-settings`의 `BaseSettings`를 사용한다. 이 방식은 `.env` 파일에 있는 값을 읽어 Python 설정 객체로 만들어준다.

현재 설정 항목은 다음과 같다.

```python
DB_USER: str = "root"
DB_PASSWORD: str = "password1234"
DB_HOST: str = "localhost"
DB_PORT: str = "3306"
DB_NAME: str = "ai_health"
```

실제 DB 접속 정보는 코드에 직접 고정하기보다 `.env` 파일에서 관리하는 것이 좋다. 단, `.env`에는 비밀번호 같은 민감한 정보가 들어갈 수 있으므로 Git에 올리지 않도록 주의해야 한다.

### 3.3 pyproject.toml

`pyproject.toml`은 Python 프로젝트 설정 파일이다.

현재 프로젝트에서는 다음 내용을 관리한다.

- 프로젝트 이름
- 프로젝트 버전
- Python 버전 조건
- 사용하는 패키지 의존성

현재 주요 의존성은 다음과 같다.

```text
fastapi[standard]
sqlalchemy[asyncio]
asyncmy
alembic
pydantic
pydantic-settings
uuid6
```

즉, 이 프로젝트는 FastAPI, 비동기 SQLAlchemy, MySQL 드라이버, Alembic 마이그레이션을 사용하는 백엔드 프로젝트다.

### 3.4 uv.lock

`uv.lock`은 `uv`가 생성하는 의존성 잠금 파일이다.

`pyproject.toml`이 필요한 패키지의 범위를 적는 파일이라면, `uv.lock`은 실제로 설치된 정확한 패키지 버전을 기록한다. 팀원이 같은 프로젝트를 설치할 때 동일한 버전의 패키지를 사용할 수 있게 해준다.

보통 직접 수정하지 않고, 패키지 추가나 업데이트를 하면 `uv`가 자동으로 수정한다.

## 4. 데이터베이스 연결 설정

DB 연결 설정은 `app/core/db/databases.py`에서 담당한다.

현재 구조는 다음과 같다.

```python
DATABASE_PREFIX = "mysql+asyncmy://"
DATABASE_URI = f"{settings.DB_USER}:{settings.DB_PASSWORD}@{settings.DB_HOST}:{settings.DB_PORT}/{settings.DB_NAME}"
DATABASE_URL = f"{DATABASE_PREFIX}{DATABASE_URI}"
```

즉, `.env` 또는 기본 설정에서 읽은 DB 정보를 조합해서 MySQL 접속 URL을 만든다.

이후 SQLAlchemy 비동기 엔진과 세션 팩토리를 생성한다.

```python
async_engine = create_async_engine(DATABASE_URL, echo=False, future=True)
AsyncSessionLocal = async_sessionmaker(
    bind=async_engine,
    autoflush=False,
    expire_on_commit=False,
)
Base = declarative_base()
```

API에서 DB 세션이 필요할 때는 다음 dependency를 사용한다.

```python
async def async_get_db():
    async with AsyncSessionLocal() as db:
        yield db
```

API 함수에서는 다음처럼 사용할 수 있다.

```python
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession

from app.core.db.databases import async_get_db


async def get_users(db: AsyncSession = Depends(async_get_db)):
    ...
```

## 5. SQLAlchemy 모델과 Alembic 마이그레이션

### 5.1 SQLAlchemy 모델 작성

SQLAlchemy 모델은 데이터베이스 테이블을 Python 클래스로 표현한다.

현재 프로젝트에는 공통으로 사용할 수 있는 mixin이 `app/core/db/models.py`에 정의되어 있다.

- `UUIDMixin`: UUID primary key 제공
- `TimestampMixin`: 생성일, 수정일 컬럼 제공
- `SoftDeleteMixin`: 논리 삭제 컬럼 제공

예를 들어 UUID 기반 회원 모델은 다음과 같이 작성할 수 있다.

```python
from sqlalchemy import String, Integer
from sqlalchemy.orm import Mapped, mapped_column

from app.core.db.databases import Base
from app.core.db.models import UUIDMixin, TimestampMixin, SoftDeleteMixin


class User(Base, UUIDMixin, TimestampMixin, SoftDeleteMixin):
    __tablename__ = "users"

    name: Mapped[str] = mapped_column(String(30), nullable=False)
    age: Mapped[int] = mapped_column(Integer, nullable=False)
    email: Mapped[str] = mapped_column(String(100), unique=True, nullable=False)
    password: Mapped[str] = mapped_column(String(255), nullable=False)
```

### 5.2 Alembic이 모델을 인식하게 하기

Alembic은 `Base.metadata`를 기준으로 모델 변경사항을 감지한다. 현재 `alembic/env.py`는 다음 설정을 사용한다.

```python
from app.core.db.databases import Base, DATABASE_URL

target_metadata = Base.metadata
```

모델 파일을 새로 만들었다면 `app/models/__init__.py`에서 import 해야 한다.

```python
from app.models.user import User
```

그래야 Alembic 실행 시 해당 모델 클래스가 로드되고, `Base.metadata`에 테이블 정보가 등록된다.

### 5.3 마이그레이션 생성과 적용

모델 작성 후 다음 명령으로 migration 파일을 생성한다.

```bash
cd AH_web_development_assignment
uv run alembic revision --autogenerate -m "create users table"
```

생성된 migration 파일을 확인한 뒤 DB에 적용한다.

```bash
uv run alembic upgrade head
```

전체 흐름은 다음과 같다.

```text
모델 작성
→ app/models/__init__.py에 import
→ alembic revision --autogenerate
→ migration 파일 확인
→ alembic upgrade head
→ DB 테이블 생성 확인
```

## 6. API 구현 흐름

API 구현은 다음 흐름으로 나누면 이해하기 쉽다.

```text
Client
→ app/apis/
→ app/services/
→ app/repositories/
→ app/models/
→ Database
```

각 계층의 책임은 다음과 같다.

| 계층 | 역할 |
|---|---|
| `apis` | URL, HTTP method, 요청/응답 연결 |
| `schemas` | 요청/응답 데이터 검증과 형태 정의 |
| `services` | 비즈니스 로직 처리 |
| `repositories` | DB query 작성과 실행 |
| `models` | DB 테이블 구조 정의 |
| `core/db` | DB 연결과 세션 관리 |

예를 들어 회원 생성 API 흐름은 다음과 같다.

```text
POST /api/v1/users
→ UserCreate schema로 요청 데이터 검증
→ service에서 이메일 중복, 비밀번호 정책 확인
→ repository에서 User 모델을 사용해 DB 저장
→ UserResponse schema 형태로 응답
```

이렇게 나누면 API가 많아져도 각 파일의 책임이 분명해진다.

## 7. PR을 거쳐 main branch에 머지한다는 의미

과제에서 "PR 과정을 거쳐 main branch에 머지해야 함"이라고 하면, `main` 브랜치에 직접 작업하지 말고 작업용 브랜치를 만든 뒤 Pull Request를 통해 병합하라는 뜻이다.

기본 흐름은 다음과 같다.

```bash
git switch main
git pull origin main
git switch -c feature/practice-api
```

작업 후에는 다음 순서로 진행한다.

```bash
git status
git add .
git commit -m "feat: practice API 구현"
git push -u origin feature/practice-api
```

그다음 GitHub에서 Pull Request를 만든다.

```text
base: main
compare: feature/practice-api
```

리뷰와 확인이 끝나면 PR을 merge해서 `main`에 반영한다.

전체 그림은 다음과 같다.

```text
main
└── feature/practice-api
    └── Pull Request
        └── main에 merge
```

## 8. 현재 프로젝트에서 주의할 점

현재 `app/apis/practice_apis.py`는 연습용 API 파일이며 내부에서 `app = FastAPI()`를 새로 생성하고 있다. 하지만 실제 프로젝트의 시작점은 `app/main.py`이므로, 이 API들은 현재 메인 앱에 자동으로 연결되지 않는다.

프로젝트 구조에 맞게 확장하려면 다음 방향이 좋다.

1. `practice_apis.py`의 `app = FastAPI()`를 `router = APIRouter()`로 변경한다.
2. API 경로를 router에 등록한다.
3. `app/main.py`에서 `app.include_router(router)`로 연결한다.
4. 요청/응답 schema는 `app/schemas/`로 분리한다.
5. DB 로직은 `app/repositories/`로 분리한다.
6. 서비스 규칙은 `app/services/`로 분리한다.

## 9. 정리

이번 프로젝트는 FastAPI 앱을 계층별로 나누어 관리하는 구조다.

- `core`는 설정과 DB 연결을 담당한다.
- `models`는 DB 테이블 구조를 담당한다.
- `schemas`는 요청과 응답의 데이터 구조를 담당한다.
- `repositories`는 DB 접근을 담당한다.
- `services`는 비즈니스 로직을 담당한다.
- `apis`는 HTTP API 라우팅을 담당한다.
- `main.py`는 FastAPI 앱을 생성하고 라우터와 정적 파일을 연결하는 시작점이다.
- `pyproject.toml`은 프로젝트 의존성을 관리한다.
- `uv.lock`은 정확한 패키지 버전을 고정한다.
- SQLAlchemy로 모델을 작성하고 Alembic으로 migration을 생성한 뒤 DB에 적용한다.
- 작업은 `main`에서 새 브랜치를 만들고 PR을 통해 다시 `main`에 병합한다.

이 흐름을 따르면 파일의 역할이 분명해지고, 팀원이 함께 작업해도 변경사항을 추적하고 검토하기 쉬워진다.
