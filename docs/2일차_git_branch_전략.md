# Git & GitHub Branch 전략

## 1. Branch란?

Branch는 기존 코드를 유지하면서 새로운 기능이나 수정 작업을 독립적으로 진행할 수 있도록 만든 작업 공간이다.

## 2. Git Flow

### Git Flow의 주요 브랜치

- main: 실제 배포 가능한 안정적인 코드
- develop: 다음 배포를 위해 개발 내용을 합치는 브랜치
- feature: 새로운 기능을 개발하는 브랜치
- release: 배포 전 최종 점검을 진행하는 브랜치
- hotfix: 운영 중 발생한 긴급 오류를 수정하는 브랜치

### Git Flow의 진행 방식

1. develop 브랜치에서 feature 브랜치를 생성한다.
2. feature 브랜치에서 기능을 개발한다.
3. 개발이 끝나면 develop 브랜치에 병합한다.
4. 배포 준비가 끝나면 release 브랜치를 만든다.
5. 검증이 끝난 코드를 main 브랜치에 병합한다.
6. 운영 중 긴급한 문제가 생기면 hotfix 브랜치를 사용한다.

### Git Flow의 장점

- 브랜치별 역할이 명확하다.
- 개발과 배포 과정을 체계적으로 관리할 수 있다.
- 여러 버전을 동시에 관리하기 좋다.

### Git Flow의 단점

- 관리해야 하는 브랜치가 많다.
- 작은 프로젝트에서는 과정이 복잡하게 느껴질 수 있다.
- 빠른 배포가 필요한 환경에서는 속도가 느려질 수 있다.

## 3. GitHub Flow

### GitHub Flow의 주요 브랜치

GitHub Flow는 기본적으로 main 브랜치와 작업 브랜치를 사용한다.

- main: 항상 배포 가능한 안정적인 코드
- 작업 브랜치: 기능 추가, 오류 수정 등 특정 작업을 진행하는 브랜치

### GitHub Flow의 진행 방식

1. 최신 main 브랜치에서 새로운 작업 브랜치를 만든다.
2. 작업 브랜치에서 코드를 작성하고 커밋한다.
3. 작업 브랜치를 GitHub에 push한다.
4. Pull Request를 생성한다.
5. 코드 리뷰가 끝나면 main 브랜치에 병합한다.
6. 병합된 main 브랜치를 배포한다.

### GitHub Flow의 장점

- 구조가 단순하고 이해하기 쉽다.
- Pull Request를 통해 코드 리뷰를 진행할 수 있다.
- 빠르고 지속적인 배포에 적합하다.

### GitHub Flow의 단점

- main 브랜치를 항상 안정적인 상태로 유지해야 한다.
- 체계적인 배포 버전 관리가 필요한 대규모 프로젝트에는 부족할 수 있다.
- 테스트와 코드 리뷰가 제대로 이루어지지 않으면 문제가 main에 반영될 수 있다.

## 4. Git Flow와 GitHub Flow 비교

| 구분 | Git Flow | GitHub Flow |
| --- | --- | --- |
| 브랜치 구조 | 여러 종류의 브랜치 사용 | main과 작업 브랜치 중심 |
| 복잡도 | 비교적 복잡함 | 비교적 단순함 |
| 배포 방식 | 정기적인 버전 배포에 적합 | 빠르고 지속적인 배포에 적합 |
| 적합한 프로젝트 | 규모가 크고 버전 관리가 중요한 프로젝트 | 웹 서비스와 빠른 개발이 필요한 프로젝트 |
| 코드 리뷰 | 팀 규칙에 따라 진행 | Pull Request 중심 |

## 5. 우리 팀에 적합한 전략

우리 팀 프로젝트에는 GitHub Flow가 더 적합하다고 생각한다.  
GitHub Flow는 구조가 단순하여 팀원들이 쉽게 이해할 수 있고, 기능별 브랜치를 만들어 동시에 작업하기 좋다. 또한 Pull Request를 통해 작업 내용을 확인한 후 main 브랜치에 병합할 수 있어 협업 과정에서 발생할 수 있는 충돌과 실수를 줄일 수 있다.

우리 팀에서는 다음과 같은 방식으로 사용할 수 있다.

1. 작업을 시작하기 전에 main 브랜치를 최신 상태로 만든다.
2. 기능이나 문서 작업별로 새로운 브랜치를 생성한다.
3. 작업이 끝나면 커밋하고 원격 저장소에 push한다.
4. Pull Request를 생성해 팀원의 확인을 받는다.
5. 검토가 끝나면 main 브랜치에 병합한다.

## 6. 학습 회고

이번 학습을 통해 브랜치는 단순히 코드를 복사하는 기능이 아니라, 여러 개발자가 서로의 작업에 영향을 주지 않고 협업하기 위한 작업 공간이라는 것을 알게 되었다. Git Flow는 체계적인 배포 관리에 유리하고, GitHub Flow는 단순하고 빠른 협업에 적합하다는 차이도 이해할 수 있었다. 앞으로 팀 프로젝트에서는 작업 목적에 맞게 브랜치를 나누고 Pull Request를 활용해야겠다.
